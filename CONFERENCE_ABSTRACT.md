## 1. АКТУАЛЬНІСТЬ ДОСЛІДЖЕННЯ (400 слів)

### Проблемна область

Сучасні корпоративні мережи генерують мільйони рядків діагностичних та конфігураційних даних щодня. Традиційні методи аналізу передбачають:

1. **Ручний розбір логів** – адміністратор витрачає годинами на ручне прочитання (неприйнято трудомісткого, схильне до помилок, неможливе для >50 пристроїв)
2. **Універсальні комерційні рішення** (Splunk, NetFlow) – непомірно дорогі ($500–$5000+/рік), складні у розгортанні
3. **Open-source рішення** (ELK Stack) – не спеціалізовані на VRP, потребують значних знань налаштування

### Актуальність чотирьох вимірів

**Безпека** (HIGH): Неправильна конфігурація створює вразливості. Виявлення помилок ACL, BGP, паролів має відбуватися автоматично.

**Економічність** (HIGH): Адміністратори витрачають 80–90% часу на аналіз. Один спеціаліст обслуговує лише 5–10 пристроїв. Вільне рішення заощадить мільйони гривень на ліцензіях.

**Масштабованість** (MEDIUM): 5G, IoT генерують експоненціально більше логів. Потреба в легких рішеннях для обладнання з обмеженими ресурсами.

**Прозорість** (MEDIUM): Open-source дозволяє розуміти логіку алгоритмів та правил.

### Мета та завдання

**Мета**: Розробити спеціалізований автономний софтвер для автоматизованого аналізу логів з видобутком даних та виявленням загроз.

**Завдання**:
1. Розробити евристичні алгоритми парсингу VRP (≥95% вилучення)
2. Реалізувати 70+ правил виявлення аномалій (100% точність на реальних сітях)
3. Розробити К-means кластеризацію інтерфейсів
4. Оптимізувати для низькопродуктивного обладнання (4 ГБ RAM, потокове читання O(1) пам'яті)
5. Забезпечити мульти-форматний експорт (Excel, JSON, CSV, DOT)
6. Випустити як open-source з MIT ліцензією

## 3. ТЕОРЕТИЧНІ ОСНОВИ (450 слів)

### 3.1 Евристичний парсинг

Метод видобутку структурованих даних із напівструктурованого тексту:

```
Парсер = Regex + Контекстний аналіз + Резильйентність

Приклад (VRP інтерфейс):
interface GigabitEthernet0/5/11
 description Uplink
 ip address 192.0.2.1 255.255.255.0

Крок 1: Regex: interface (\S+) → Ім'я
Крок 2: Контекст: вкладені лінії належать інтерфейсу
Крок 3: Резильйентність: продовжити при варіаціях версії
```

### 3.2 Виявлення аномалій (70+ правил)

**Аномалія** – конфігурація, що відхиляється від норми.

```
RULE: Interface_Without_ACL
IF (Interface.HasIP) AND (Device.ACLs.Count == 0)
THEN Anomaly(Type="Security", Severity="High", ...)
```

**Категорії**:
- **Безпека** (30): ACL, пароль, BGP, VPN, IP
- **Продуктивність** (20): CPU, пам'ять, помилки
- **Конфігурація** (20): NTP, таймаути, версія

### 3.3 Кластеризація К-means

$$U_{max} = \max(U_{in}, U_{out})$$

**5 кластерів**:
- High: $U > 80\%$
- Medium: $50\% ≤ U ≤ 80\%$
- Low: $U < 50\%$
- Error-Prone: помилки >100
- Shutdown: адміністративно вимкнено

### 3.4 Алгоритмічна складність

| Операція | Часова | Просторова |
|----------|--------|-----------|
| Читання (потокове) | $O(n)$ | $O(1)$ |
| Парсинг | $O(n \cdot m)$ | $O(m)$ |
| Аномалії | $O(i \cdot r)$ | $O(r)$ |
| **ЗАГАЛОМ** | **$O(n + r)$** | **$O(1) + O(res)$** |
